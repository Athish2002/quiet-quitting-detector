# /stride-threat-model Skill Instructions

When this skill is invoked, execute the following steps **in order**.

---

## Step 1 – Analyse the Project Architecture

Read the following source files to understand the system fully before writing anything:

- `src/orchestrator_agent.py`   — pipeline entry point, CSV reading, employee routing
- `src/trend_detector_agent.py` — programmatic signal detection + LLM enrichment
- `src/risk_scorer_agent.py`    — memory load, LLM scoring, recurrence bonus, save to disk
- `src/manager_briefing_agent.py` — briefing generation for Watch / At Risk / Silent Exit
- `src/agent.py`                — root agent wiring and App object
- `CONTEXT.md`                  — the six strict project rules governing all agent behaviour
- `data/memory/`                — persisted risk JSON files (check for any existing files)

---

## Step 2 – Assess Exactly These Four Threats

Evaluate each threat in the context of the actual code you read in Step 1.
Do **not** genericise. Reference specific functions, files, and line-level behaviours.

### Threat 1 — False Positive Risk
**Definition:** A high-performing employee who is having a genuinely difficult personal week
(bereavement, illness, family crisis) could have depressed metrics for a short period and
get flagged as disengaged, triggering an unnecessary manager intervention.

Assess:
- Does the trend detector require 2+ consecutive weeks before raising a signal?
- Is there a single-week spike filter?
- Does the risk scorer include any dampening for isolated dips?
- Does the briefing agent's tone reduce or amplify this risk?

### Threat 2 — MCP / Web Search Poisoning
**Definition:** If a web-search MCP tool is added to any agent in the future, a poisoned
or adversarially crafted search result could inject harmful re-engagement advice into a
manager briefing (e.g. surveillance recommendations, punitive actions disguised as support).

Assess:
- Are there currently any web tools registered on any of the four agents?
- Do the system instructions include explicit constraints that would block harmful advice
  even if a poisoned search result were fed to the model?
- Is there any output validation layer?

### Threat 3 — Memory Drift
**Definition:** Stale or incorrect risk scores persisted in `data/memory/` could unfairly
anchor the risk scorer's evaluation of a now-recovered employee, permanently biassing their
score upward even after they return to healthy engagement.

Assess:
- How many prior weeks does `score_risk()` load?
- What is the maximum memory lookback window?
- Is there a decay or expiry mechanism for old scores?
- What happens if a corrupted or manually edited JSON file is in `data/memory/`?

### Threat 4 — Privacy Breach
**Definition:** Employee first names and numeric risk scores could leak into Python
tracebacks, `print()` statements, log files, or error messages, violating CONTEXT Rule 1
and potentially exposing sensitive HR information to server logs.

Assess:
- Every `print()` call in `orchestrator_agent.py` — does it include first names or scores?
- Exception handlers — do any `except Exception as e:` blocks propagate `e` to output?
- Do session IDs include first names (e.g. `session_{first_name_lower}_risk`)?
- Does `fast_api_app.py` emit any structured logs that could include agent session state?

---

## Step 3 – Write One Concrete Mitigation Per Threat

Each mitigation must:
- Be specific to **this codebase** (reference actual files, functions, or configuration)
- Be implementable in the next sprint
- Not contradict CONTEXT.md rules

---

## Step 4 – Assign an Overall Privacy Risk Rating

Choose one of: **Low**, **Medium**, or **High**

Justify the rating based on:
- The number and severity of unmitigated threats found
- The sensitivity of the data (employee engagement metrics, HR-adjacent outputs)
- Whether personally identifiable signals (first names + scores) can currently escape the system boundary
