# STRIDE Threat Model — Quiet Quitting Detector

> Generated by the `/stride-threat-model` skill.
> Date: 2026-06-29

## Architecture Summary

The Quiet Quitting Detector is a four-agent ADK 2.0 pipeline. The **Orchestrator** reads weekly employee CSV files, groups them by first name, and calls the other three agents in sequence for each employee. The **Trend Detector** applies rule-based signal detection against each employee's own week-1 baseline and uses an LLM to enrich confirmed (2+ consecutive week) signals. The **Risk Scorer** loads all prior memory files from `data\memory\`, queries a Gemini LLM for a disengagement score (1–10), applies a recurrence bonus, and persists the result. The **Manager Briefing Agent** generates a warm HR-safe briefing only for employees classified as Watch, At Risk, or Silent Exit.

## Threat 1: False Positive Risk

**STRIDE Category:** Repudiation / Elevation of Privilege (agent over-reach)
**Affected Components:** `trend_detector_agent.py`, `risk_scorer_agent.py`, `manager_briefing_agent.py`

### Finding

The trend detector (`_require_consecutive()`, lines 119–153) correctly requires a signal to appear in **2 or more consecutive weeks** before it is confirmed. A single bad week is explicitly filtered out. The week-1 baseline is the employee's own historical baseline (`_get_baseline()`, lines 57–65), not a cohort average, so an inherently low performer is not unfairly compared against top performers.

However, two gaps remain:

1. **No minimum signal count before a briefing is triggered.** A single confirmed signal (e.g. 2 consecutive weeks of slightly elevated after-hours logins — possibly due to a project deadline) is sufficient to push an employee to `Watch` (score ≥ 4) and generate a manager briefing. There is no lower threshold that requires, say, 2 distinct signal types before escalation.

2. **The recurrence bonus in `risk_scorer_agent.py` (lines 207–217) applies a hard +1 to the score** whenever an employee has been Watch-or-above for 2+ prior weeks — even if they recovered last week. This means an employee who had 2 hard weeks months ago and has since fully recovered could still receive a permanent +1 inflation on every subsequent score. There is no decay mechanism for the recurrence bonus.

3. **The manager briefing fallback (line 79–83)** includes the employee's first name in the error string, and the briefing prompt (lines 46–53) passes the numeric score directly to the LLM — creating a low-level paper trail that correlates names to risk scores even in error paths.

### Mitigation

Introduce a **minimum signal-count gate** in `orchestrator_agent.py` before calling `generate_briefing()`: only trigger a briefing if 2 or more distinct signal types are confirmed AND the score is ≥ 5 (`Watch`). Additionally, add a **recurrence decay** in `risk_scorer_agent._count_elevated_history_weeks()`: only count prior elevated weeks that fall within the last 6 weeks (i.e. set `MAX_HISTORY_WEEKS = 6` and slice `history[-6:]`) so that historical patterns older than six weeks do not continue to penalise a recovered employee.

## Threat 2: MCP / Web Search Poisoning

**STRIDE Category:** Tampering / Spoofing
**Affected Components:** All four agents (potential future tool registration)

### Finding

**No web tools are currently registered** on any of the four ADK agents. All four agents are instantiated with `Agent(name=..., model=Gemini(...), instruction=...)` and no `tools=[...]` parameter — confirmed across `trend_detector_agent.py`, `risk_scorer_agent.py`, `manager_briefing_agent.py`, and `orchestrator_agent.py`. There is no MCP server wiring in `agents-cli-manifest.yaml` or `src/agent.py`.

The risk is therefore **latent but high** for the future: the system prompt instructions in each agent do contain natural language prohibitions against disciplinary action and harmful advice, but there is **no programmatic output validation layer** — the LLM's text response is accepted verbatim and returned as the briefing. If a future developer adds a web-search tool (for re-engagement research), a poisoned result containing phrasing like "performance improvement plan" or "monitoring" could bypass the system prompt guardrails because LLMs are susceptible to prompt injection via tool output.

The `fast_api_app.py` already wires OpenTelemetry tracing and a Google Cloud Logging client (`logger = logging_client.logger(__name__)`). Any future agent response containing injected content would be emitted to Cloud Logging unfiltered.

### Mitigation

Implement a **post-generation output validator** as a Python function called before any briefing text is returned to the orchestrator. The validator should scan for a deny-list of HR-unsafe phrases (e.g. `"performance improvement"`, `"disciplinary"`, `"monitored"`, `"logged"`, `"surveillance"`, `"terminated"`, `"consequence"`) using a case-insensitive regex check. If any deny-list term is detected, the function should replace the entire response with the safe fallback briefing string already defined in `generate_briefing()`'s `except` block, and emit a warning to the internal logger (without including the employee name). Add this validator to `manager_briefing_agent.py` as `_validate_briefing(text: str) -> str` and call it immediately before `return response_text.strip()` on line 75.

## Threat 3: Memory Drift

**STRIDE Category:** Tampering / Information Disclosure
**Affected Components:** `risk_scorer_agent.py`, `data/memory/`

### Finding

`_load_employee_history()` (lines 70–95) uses `glob.glob()` to load **all** previous JSON files matching `{firstname}_week*.json` — there is **no maximum lookback window**. In a production system running 52 weeks per year, all 52 prior files would be loaded, summarised into the LLM prompt, and the `elevated_weeks` counter would accumulate indefinitely. Three specific drift risks exist:

1. **Unbounded accumulation:** An employee who was `At Risk` for 3 weeks two years ago but has been `Healthy` ever since would still have `elevated_weeks >= 2`, triggering the permanent +1 recurrence bonus on line 209 every week forever.

2. **Corrupted or manually edited files:** The `except Exception: continue` in `_load_employee_history()` silently skips unreadable files. However, a **syntactically valid but semantically corrupted** JSON file (e.g. `"score": 9` manually inserted by a bad actor with file system access) would be loaded and included in the history summary without any integrity check. There is no checksum or audit trail for memory files.

3. **No retirement of stale scores:** `data\memory\` is a flat directory. Files accumulate without any archival, rotation, or TTL policy. If an employee leaves the company, their files persist indefinitely and could be read against a future employee who happens to share the same first name.

### Mitigation

Add a `MAX_HISTORY_WEEKS = 8` constant to `risk_scorer_agent.py` and modify `_load_employee_history()` to slice `matched_files[-MAX_HISTORY_WEEKS:]` after sorting — ensuring only the 8 most recent files are ever loaded. For the name-collision edge case, add a **file-creation date check**: only load files created in the current calendar year (`os.path.getmtime(file_path)` compared to `datetime.date.today().year`). Document both constants at the top of the file with a comment explaining the rationale, so future maintainers do not remove them without understanding the privacy implication.

## Threat 4: Privacy Breach

**STRIDE Category:** Information Disclosure
**Affected Components:** `orchestrator_agent.py`, `fast_api_app.py`, session IDs

### Finding

Four concrete privacy leak surfaces exist in the current codebase:

1. **`print()` statements with file paths (orchestrator_agent.py, lines 122, 126, 48, 55):** `print(f"Error reading CSV file {file_path}: {e}")` emits the CSV file path to stdout. On Windows, this path is `data\weekly\week_1.csv` (no names), which is acceptable — but `print(f"Data folder '{weekly_folder}' does not exist.")` and `print(error_msg)` at lines 48–56 would also surface to any server log aggregator capturing stdout.

2. **Session IDs embed first names:** All four agents construct session IDs of the form `session_{first_name_lower}_risk`, `session_{first_name_lower}_trends`, `session_{first_name_lower}_briefing`. ADK's `InMemoryRunner` stores these in-process, so they do not persist. However, if the project is migrated to a Cloud SQL or persistent session backend (configured via `SESSION_SERVICE_URI` in `app_utils/services.py`), session IDs containing first names would be persisted to the session store and visible in database queries.

3. **`fast_api_app.py` Google Cloud Logging (line 44):** `logger = logging_client.logger(__name__)` initialises a Cloud Logging sink. The `get_fast_api_app()` call on line 79 uses ADK's built-in FastAPI wrapper, which may emit agent event streams — including session IDs and intermediate LLM prompts that contain first names — to this Cloud Logging client if `otel_to_cloud=True` were ever changed.

4. **Manager briefing error fallback (manager_briefing_agent.py, lines 79–83):** The `except` block in `generate_briefing()` returns a string that begins `f"Manager Briefing for {first_name}:"` — embedding the employee's first name directly in the error message that is stored in `pipeline_results[first_name]["briefing"]` and then serialised into the LLM prompt sent to the orchestrator (orchestrator_agent.py, line 190: `json.dumps(pipeline_results, indent=2)`). This means a first name associated with an error event is sent to the Gemini API, potentially included in API request logs.

### Mitigation

Replace all session IDs with **hashed pseudonyms**: instead of `f"session_{first_name_lower}_risk"`, use `f"session_{hashlib.sha256(first_name_lower.encode()).hexdigest()[:12]}_risk"`. This prevents first names from appearing in session stores, telemetry traces, or API request metadata while still allowing per-employee session isolation. Apply this change across all four agent files. Additionally, update the manager briefing error fallback to remove the first name from the returned string: replace `f"Manager Briefing for {first_name}:"` with the generic `"Manager Briefing:"` — the calling code in the orchestrator already knows which employee the briefing belongs to from the `pipeline_results` dictionary key.

## Overall Privacy Risk Rating

**Rating: Medium**

The system has meaningful privacy protections in place: it uses first names only (never surnames or IDs), all exception handlers suppress raw API errors, and the trend detector requires 2+ consecutive weeks before raising a signal — reducing noise-driven escalations. However, two unmitigated issues prevent a Low rating: first-name leakage into session IDs and error message strings creates real exposure if the system is connected to a persistent session backend or Cloud Logging, and the unbounded memory lookback in the risk scorer means historical employee data accumulates without expiry, creating a long-term data minimisation violation. None of the threats identified are exploitable remotely without file system access or API credential access, which is why the rating does not reach High.
